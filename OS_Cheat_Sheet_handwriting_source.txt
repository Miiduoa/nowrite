==== L5 В· Threads еҹ·иЎҢз·’ ====

5.1 Thread vs Process еҹ·иЎҢз·’ vs зЁӢеәҸ
В· Definition вҖ” Process: independent program in execution / Thread: lightweight unit within a process
В· Memory вҖ” Process: separate address space / Thread: shares code/data/files
В· Resources вҖ” Process: code,data,files,registers,stack / Thread: only thread ID, registers, stack, PC
В· Creation вҖ” Process: heavy (expensive) / Thread: light (cheap еҝ«йҖҹ)
В· Ctx switch вҖ” Process: slow / Thread: fast
В· Comm. вҖ” Process: IPC (pipes, sockets) / Thread: shared memory directly

5.2 Multithreading Benefits еҘҪиҷ•
В· Responsiveness еӣһжҮү: UI responsive while bg work runs
В· Resource Sharing е…ұдә«: share code/data/files вҶ’ less memory
В· Economy з¶“жҝҹ: cheaper than creating processes
В· Scalability ж“ҙеұ•: run on multiple cores in parallel
В· Challenges: divide tasks, balance load, data splitting/dependency, testing

5.3 Threading Models еҹ·иЎҢз·’жЁЎеһӢ
В· User дҪҝз”ЁиҖ…: user-level lib (Pthreads, Java, Win32). Kernel ж ёеҝғ: by OS kernel
В· Many-to-One еӨҡе°ҚдёҖ (many userвҶ’1 kernel): efficient; one blockвҶ’ALL block, no parallelism. Green threads
В· One-to-One дёҖе°ҚдёҖ (each userвҶ’1 kernel): more concurrency; thread=kernel thread (overhead). Windows, Linux
В· Many-to-Many еӨҡе°ҚеӨҡ (manyвҶ’вүӨ many): best flexibility, true parallelism; complex. Solaris

5.4 Pthread API еҮҪејҸеә« (POSIX thread lib for C/C++ on Unix/Linux)
В· pthread_create: create е»әз«Ӣ (&tid, attr, routine, arg)
В· pthread_join: wait to finish зӯүеҫ…; gets return value
В· pthread_detach: detach еҲҶйӣў; auto-free, can't join later
В· pthread_exit: terminate self иҮӘиЎҢзөҗжқҹ; return status
В· Flow: create() вҶ’ worker runs вҶ’ exit() вҶ’ main join() collects

5.5 Java & Linux Threads
В· Java: extend Thread OR implement Runnable; native OS threads (WinвҶ’Win32 1:1, SolarisвҶ’Pthreads M:M)
В· Linux: fork()=new process (copy); clone()=thread-like (shares memory)
В· clone flags вҖ” CLONE_FS: file-system info / CLONE_VM: memory space / CLONE_SIGHAND: signal handlers / CLONE_FILES: open files

5.6 Threading Issues иӯ°йЎҢ
В· fork()&exec(): if exec() follows вҶ’ dup only caller (efficient); exec() replaces process, all threads lost
В· Cancellation еҸ–ж¶Ҳ: async=stop now (dangerous); deferred (default)=check at cancel points (safer)
В· Signal дҝЎиҷҹ: sync=self (div-by-0); async=external (Ctrl+C) вҶ’ specific/broadcast/handler thread
В· Thread Pool жұ : pre-create & reuse; faster, caps count. Size вҶҗ cores, load, memory
В· Thread-Specific: private data per thread (own copy)

==== L6 В· CPU Scheduling жҺ’зЁӢ ====

6.1 Concepts & Metrics еҹәжң¬жҰӮеҝө
В· CPU burst зҲҶзҷј / I/O burst (waiting I/O)
В· CPU-bound еҜҶйӣҶ: long CPU, few I/O. I/O-bound: short CPU, frequent I/O (interactive)
В· Transitions: 1 RunningвҶ’Waiting (I/O) = non-preempt; 2 RunningвҶ’Ready (interrupt) = preempt; 3 WaitingвҶ’Ready (I/O done) = preempt; 4 Terminates = non-preempt
В· Non-preempt йқһжҗ¶дҪ”: only 1 & 4. Preempt жҗ¶дҪ”: 1-4, need sync!
В· Metrics вҖ” Turnaround е®ҢжҲҗ = Completion вҲ’ Arrival (Min); Waiting зӯүеҫ… = Turnaround вҲ’ Burst (Min); Response еӣһжҮү = First Sched вҲ’ Arrival (Min); Throughput з”ўеҮә = procs done / time (Max); Utilization = % CPU busy (Max)
В· Dispatcher latency = stop one + start another (keep min)

6.2 FCFS е…ҲдҫҶе…ҲжңҚеӢҷ
В· Non-preemptive; runs until done
В· Convoy Effect иӯ·йҖҒ: short jobs stuck behind long вҶ’ long avg wait
В· Arrival order changes waiting times a lot

6.3 SJF / SRTF жңҖзҹӯе·ҘдҪңе„Әе…Ҳ
В· SJF (non-preempt): shortest next burst, run to completion
В· SRTF (preempt) жңҖзҹӯеү©йӨҳ: shorter arrival вҶ’ preempt
В· OPTIMAL min avg waiting time; can't know burst вҶ’ predict
В· П„(n+1) = ОұВ·t(n) + (1вҲ’Оұ)В·П„(n);  ОұвүҲ0.5, t=last burst, П„=last predict

6.4 Priority е„Әе…Ҳж¬Ҡ
В· Highest priority wins; preempt or non-preempt
В· SJF = priority where priority = burst
В· Starvation йЈўйӨ“: low-pri never run вҶ’ Aging иҖҒеҢ– raises priority over time

6.5 Round Robin (RR) ијӘиҪү
В· Fixed TQ жҷӮй–“йҮҸеӯҗ (10-100ms); after TQ вҶ’ end of queue
В· TQ large вҶ’ FCFS; TQ small вҶ’ too many ctx switches
В· Time-sharing & fairness; better response, worse turnaround than SJF
В· Rule: 80% of bursts < TQ

6.6 Multilevel Queue еӨҡеұӨдҪҮеҲ—
В· Split into queues (system, interactive, batch, student); each own algorithm
В· Process stays in queue permanently (no move)
В· Between: fixed priority (starvation) OR time slice (80%fg / 20%bg)
В· Order: system > interactive > editing > batch > student. I/O-bound вҶ’ higher

6.7 MLFQ еӨҡеұӨеӣһйҘӢ
В· Like MLQ but CAN MOVE between queues; Aging prevents starvation
В· Q0 high = 8ms (new jobs here); Q1 = 16ms (not done in Q0 вҶ’ here); Q2 low = вҲһ (runs when Q0,Q1 empty)
В· Config: #queues, algo/queue, promote еҚҮзҙҡ / demote йҷҚзҙҡ

6.8 Multi-Processor / Core
В· Multi-Processor еӨҡиҷ•зҗҶеҷЁ:
  - Asymmetric йқһе°ҚзЁұ: 1 CPU does OS, others user code
  - SMP е°ҚзЁұ: each CPU self-schedules; needs sync
  - Affinity иҰӘе’ҢжҖ§: stay same CPU вҶ’ warm cache. Soft=prefer / Hard=forced
  - NUMA: CPU+mem grouped; keep near local mem
  - Load Balance: Push (busyвҶ’idle) / Pull (idleвҶҗbusy). vs affinity!
В· Multi-Core еӨҡж ёеҝғ:
  - Memory stall еҒңж»Ҝ: cache miss wastes cycles
  - Hyper-threading: 2+ HW threads/core use stall time
  - Coarse зІ— (switch on stall, flush) / Fine зҙ° (each cycle)
  - 2 levels: OSвҶ’HW thread; core picks HW thread
В· Eval вҖ” Deterministic: fixed workload, not general; Queueing: math avg; Simulation: random/real data; Real impl.: accurate, expensive

вҳ… Scheduling Comparison жҜ”ијғ
В· FCFS вҖ” Pre: N, Starve: N вҖ” convoy effect
В· SJF вҖ” Pre: N, Starve: Y вҖ” optimal avg wait
В· SRTF вҖ” Pre: Y, Starve: Y вҖ” preemptive SJF
В· Priority вҖ” Pre: Both, Starve: Y вҖ” aging fixes starve
В· RR вҖ” Pre: Y, Starve: N вҖ” TQ critical
В· MLQ вҖ” Pre: Y, Starve: Y вҖ” fixed, no move
В· MLFQ вҖ” Pre: Y, Starve: N* вҖ” *aging; moves

==== L7 В· Synchronization еҗҢжӯҘ ====

7.1 Critical Section иҮЁз•ҢеҚҖ
В· CS: code accessing shared data. Only ONE process in CS
В· do { entry-section() йҖІе…ҘеҚҖ; CRITICAL SECTION иҮЁз•ҢеҚҖ; exit-section() йӣўй–ӢеҚҖ; remainder-section() еү©йӨҳеҚҖ } while (TRUE);
В· Requirements: 1 Mutual Excl дә’ж–Ҙ (Pi in CS вҶ’ no other in CS); 2 Progress йҖІеұ• (none in CS & some want вҶ’ pick in finite time); 3 Bounded Wait жңүйҷҗ (bounded entries before a waiter, no starve!)
В· Race condition з«¶зҲӯ: result depends on order (counter++ / -- interleave)

7.2 Peterson (2 proc)
В· SW, 2 proc. Shared int turn; bool flag[2]
В· // Pi (i=0/1, j=1вҲ’i): do { flag[i]=TRUE (I want in); turn=j (let you first); while (flag[j] && turn==j) wait; CRITICAL SECTION; flag[i]=FALSE (done); remainder } while (TRUE)
В· вң“ME вң“Progress вң“Bounded
В· flag=жғійҖІеҺ», turn=и®“е°Қж–№е…Ҳ; йғҪжғійҖІжҷӮ turn жұәе®ҡиӘ°е…Ҳ

7.3 Bakery (n proc)
В· Ticket number; smallest first; tie вҶ’ lower ID. (a,b)<(c,d) if a<c or (a=c & b<d)
В· do { choosing[i]=TRUE; num[i]=max(num[0..n-1])+1 (ticket); choosing[i]=FALSE; for (j=0; j<n; j++) { while (choosing[j]); while (num[j]!=0 && (num[j],j)<(num[i],i)) }; CRITICAL SECTION; num[i]=0 (release); remainder } while (1)
В· вң“ All 3. choosing[] stops simultaneous-pick conflicts

7.4 Hardware зЎ¬й«”ж”ҜжҸҙ
В· Disabling interrupts fails on multiprocessor вҶ’ atomic instr
В· bool TestAndSet(bool &lock) { bool v=lock; lock=TRUE; return v; } // atomic дёҚеҸҜдёӯж–·
В· do { while (TestAndSet(lock)) spin; CS; lock=FALSE } while (1)
В· // Swap (shared bool lock=FALSE): do { key=TRUE; while (key==TRUE) Swap(lock,key); CS; lock=FALSE } while (1)
В· вң“ME вң“Progress вң—Bounded
В· е…©иҖ…йғҪдёҚж»ҝи¶і Bounded Wait, process еҸҜиғҪдёҖзӣҙ spin

7.5 Semaphore иҷҹиӘҢ
В· Integer via atomic wait()/signal(). Binary дәҢе…ғ (0/1)=mutex. Counting иЁҲж•ё (вүҘ0)=#units
В· wait(S) { while (S<=0) busy-wait; S-- }   signal(S) { S++ }
В· Busy waiting wastes CPU (spinlock)
В· POSIX вҖ” sem_init(s,psh,val): init with value; sem_wait(s): decrement, block if 0 (P); sem_post(s): increment (V); sem_getvalue: get current value; sem_destroy(s): destroy
В· POSIX sem вҲҲ POSIX, NOT Pthread; works with/without threads

7.6 Mutex & Condition Var.
В· pthread_mutex_init(&m,NULL) еҲқе§ӢеҢ–; pthread_mutex_lock(&m) йҖІе…Ҙ; Critical Section; pthread_mutex_unlock(&m) йӣўй–Ӣ; pthread_mutex_destroy(&m) йҠ·жҜҖ
В· CV ops вҖ” cond_wait(&cv,&m): release mutex, sleep, re-acquire; cond_signal(&cv): wake ONE; cond_broadcast: wake ALL
В· RULE: CV ops only while mutex LOCKED!
В· // Waiting: lock(&m); if (x!=0) cond_wait(c,m); unlock(&m); take_action()
В· // Signaling: lock(&m); x--; if (x==0) cond_signal(c); unlock(&m)
В· wait() дёүжӯҘ: в‘  ж”ҫйҺ–+зқЎ  в‘Ў иў«е–ҡйҶ’ (йҺ–иў«дҪ”)  в‘ў йҮҚжӢҝйҺ–+з№јзәҢ

7.7 Producer-Consumer з”ҹз”ўж¶ҲиІ»
В· Shared: buffer[], counter, in, out
В· Incorrect (entry at very start): consumer first вҶ’ deadlock!
В· Inefficient (CS wraps all): works but less concurrency
В· Efficient (CS wraps only counter++ / --): more overlap
В· еҸӘжҠҠ counter дҝ®ж”№ж”ҫиҮЁз•ҢеҚҖ, buffer ж“ҚдҪңж”ҫеӨ–йқў вҶ’ ж•ҲзҺҮжңҖеҘҪ

==== L8 В· Deadlock жӯ»зөҗ ====

8.1 Four Conditions еӣӣжўқд»¶ (all FOUR hold at once вҶ’ all wait forever)
В· 1 Mutual Excl дә’ж–Ҙ: вүҘ1 non-sharable resource
В· 2 Hold & Wait жҢҒжңүзӯүеҫ…: holds вүҘ1 AND waits more
В· 3 No Preempt дёҚеҸҜжҗ¶: released only voluntarily
В· 4 Circular еҫӘз’°: P0вҶ’P1вҶ’вҖҰвҶ’PnвҶ’P0

8.2 RAG иіҮжәҗй…ҚзҪ®ең–
В· PiвҶ’Rj = Request: Pi waits for Rj. RjвҶ’Pi = Assignment: Rj held by Pi
В· Process=в—Ӣ, Resource=в–Ў (dots=instances)
В· No cycle вҶ’ no deadlock
В· Cycle + single inst вҶ’ deadlock
В· Cycle + multi inst вҶ’ maybe

8.3 Prevention й җйҳІ (break вүҘ1 of the 4 conditions)
В· Mutual Excl: sharable (read-only); not always possible
В· Hold & Wait: request all upfront / release before new вҶ’ low util, starve
В· No Preempt: denied вҶ’ release all, restart; only if state savable
В· Circular: total order, request increasing. Most practical!

8.4 Avoidance (RAG) йҒҝе…Қ
В· Safe: a safe sequence exists (each can finish)
В· Unsafe: no safe seq вҶ’ potential (not guaranteed)
В· Unsafe вү  Deadlock; Deadlock вҠӮ Unsafe вҠӮ All
В· RAG (single inst): Claim edge (PiвҶ’Rj dashed) = future request, a priori. Request: claimвҶ’request; Release: assignвҶ’claim. Grant only if no cycle. Detect O(n^2)

8.5 Banker's Algorithm йҠҖиЎҢе®¶ (multi-instance, Dijkstra. Need = Max вҲ’ Allocation)
В· Safety: 1 Work=Available, Finish[i]=false; 2 find i: Finish[i]=F & Need[i]вүӨWork (noneвҶ’4); 3 Work+=Alloc[i], Finish[i]=T, вҶ’2; 4 all Finish=T вҶ’ SAFE вң“ else UNSAFE вң—
В· Request[i]: 1 Request>Need вҶ’ ERROR; 2 Request>Avail вҶ’ WAIT; 3 pretend AvailвҲ’=R, Alloc+=R, NeedвҲ’=R; 4 Safety? SAFEвҶ’grant; UNSAFEвҶ’rollback, WAIT

вҳ… Banker's Example зҜ„дҫӢ (Total A10 B5 C7, Available (3,3,2))
В· P0 вҖ” Max 7 5 3, Alloc 0 1 0, Need 7 4 3
В· P1 вҖ” Max 3 2 2, Alloc 2 0 0, Need 1 2 2
В· P2 вҖ” Max 9 0 2, Alloc 3 0 2, Need 6 0 0
В· P3 вҖ” Max 2 2 2, Alloc 2 1 1, Need 0 1 1
В· P4 вҖ” Max 4 3 3, Alloc 0 0 2, Need 4 3 1
В· Work=(3,3,2); P1 (1,2,2)вүӨ вң“ вҶ’(5,3,2); P3 (0,1,1)вүӨ вң“ вҶ’(7,4,3); P4 (4,3,1)вүӨ вң“ вҶ’(7,4,5); P2 (6,0,0)вүӨ вң“ вҶ’(10,4,7); P0 (7,4,3)вүӨ вң“
В· seq <P1,P3,P4,P2,P0> вң“
В· P1 asks (1,0,2): вүӨNeed вҶ’ safe вҶ’ GRANT. P4 asks (3,3,0): вҶ’ UNSAFE / DENY

8.6 Detection еҒөжё¬
В· Single: Wait-for graph (drop resources). Deadlock <=> cycle
В· Multi: Banker-like w/ Alloc, Request (not Max!), Avail. Not "finish" вҶ’ deadlocked
В· Total A7 B2 C6, Available (0,0,0)
В· P0 вҖ” Alloc 0 1 0, Request 0 0 0
В· P1 вҖ” Alloc 2 0 0, Request 2 0 2
В· P2 вҖ” Alloc 3 0 3, Request 0 0 0
В· P3 вҖ” Alloc 2 1 1, Request 1 0 0
В· P4 вҖ” Alloc 0 0 2, Request 0 0 2
В· <P0,P2,P3,P1,P4> all finish вҶ’ no deadlock. If P2 asks (0,0,1) вҶ’ P1-P4 deadlocked

8.7 Recovery жҒўеҫ©
В· Termination зөӮжӯў: kill ALL, or 1-by-1. Victim вҶҗ priority, time, resources
В· Preemption жҗ¶дҪ”: take resources; rollback to saved state; avoid same victim (starve); safe order

вҳ… Deadlock Strategy зӯ–з•Ҙ
В· Prevention й җйҳІ вҖ” cost High вҖ” break 1 of 4
В· Avoidance йҒҝе…Қ вҖ” cost Med вҖ” max-need; RAG / Banker
В· Detect + Recover вҖ” cost Low вҖ” wait-for / matrix
В· Ignore еҝҪз•Ҙ вҖ” cost None вҖ” most OSes (Win, Linux)
